    JAVA工程师（互联网时代）

    JAVA微服务工程师（云计算+大数据）（分布式+微服务）–》高并发、高可用。高性能。

    高并发高可用高性能

    高可用

    “高可用性”（High Availability）通常来描述一个系统经过专门的设计，从而减少停工时间，而保持其服务的高度可用性。(一直都能用)

    高可用：6个9 99.9999% 全年停机不超过32秒。

    高并发

    高并发（High Concurrency）是互联网分布式系统架构设计中必须考虑的因素之一，它通常是指，通过设计保证系统能够同时并行处理很多请求。

    ​ 高并发相关常用的一些指标有响应时间（Response Time），吞吐量（Throughput），每秒查询率QPS（Query Per Second），并发用户数等。

    响应时间：系统对请求做出响应的时间。例如系统处理一个HTTP请求需要200ms，这个200ms就是系统的响应时间。

    吞吐量：单位时间内处理的请求数量。

    QPS：每秒响应请求数。在互联网领域，这个指标和吞吐量区分的没有这么明显。
    并发用户数：同时承载正常使用系统功能的用户数量。例如一个即时通讯系统，同时在线量一定程度上代表了系统的并发用户数。

    提升系统的并发能力

    提高系统并发能力的方式，方法论上主要有两种：垂直扩展（Scale Up）与水平扩展（Scale Out）。

    垂直扩展

    垂直扩展：提升单机处理能力。垂直扩展的方式又有两种：
    （1）增强单机硬件性能，例如：增加CPU核数如32核，升级更好的网卡如万兆，升级更好的硬盘如SSD，扩充硬盘容量如2T，扩充系统内存如128G；
    （2）提升单机架构性能，例如：使用Cache来减少IO次数，使用异步来增加单服务吞吐量，使用无锁数据结构来减少响应时间；

    在互联网业务发展非常迅猛的早期，如果预算不是问题，强烈建议使用“增强单机硬件性能”的方式提升系统并发能力，因为这个阶段，公司的战略往往是发展业务抢时间，而“增强单机硬件性能”往往是最快的方法。

    总结：不管是提升单机硬件性能，还是提升单机架构性能，都有一个致命的不足：单机性能总是有极限的。所以互联网分布式架构设计高并发终极解决方案还是水平扩展。

    水平扩展

    水平扩展：只要增加服务器数量，就能线性扩充系统性能。水平扩展对系统架构设计是有要求的，难点在于：如何在架构各层进行可水平扩展的设计，

    高性能

    高性能（High Performance）就是指程序处理速度快，所占内存少，cpu低

    背景

    第一时期

    单一应用架构

    当网站流量很小时，只需一个应用，将所有功能都部署在一起，以减少部署节点和成本。此时，用于简化增删改查工作量的数据访问框架(ORM)是关键。

    All in One:所有的代码和业务都在一起。技术也不分层。(1996年–2003年之间)

    网站最早的初期原型。所有的代码、业务（数据库逻辑操作）都在同一个页面内。

    问题:

    1、代码不具备可维护性。

    2、容错性

    解决方案:

    1、分层开发

    2、MVC架构。（MVC 是一种使用设计创建 Web 应用程序的模式）

    3、服务和应用程序分离部署。

    特点：

    1、MVC分层开发（提高了代码的可维护性。解决了容错性问题）

    2、数据库和应用程序分离部署

    问题：

    当访问量逐渐增大，单一应用增加机器带来的加速度越来越小?

    解决方案：

    集群：同一个业务，部署在不同的服务器上。

    特点：

    解决：

    解决了高并发。支持高可用。

    问题：

    1、用户的请求该往哪个服务器发送？

    Nginx反向代理+负载均衡

    2、Session如何共享？

    SpringSession+RedisCluster(集群)

    数据库压力

    通过Nginx+Tomcat集群方案 解决高并发（应用性能有效提升）

    但是数据库的访问压力慢慢变大。

    问题：

    怎么来减轻数据库层面的访问压力？

    解决方案：

    读写分离

    主从数据库之间进行数据同步。主库（master)只负责增删改。从(slave)负责查询

    搜索引擎缓解数据库的访问压力+能力

    数据库本身大的数据量查询有效率低问题。例如：全文检索。

    例如：电商网站首页商品查询（京东、淘宝、…)主页商品查询，（搜索是非常核心的功能）。

    全文检索技术：

    Solr ElasticSearch

    引入缓存技术减轻数据库访问压力

    最佳解决方案：

    Redis/EHcache

    作用：Redis为了解决高并发访问。RedisCluster(高并发、高可用)

    1、读写性能非常高。

    2、数据类型丰富。

    3、原子性（命令）。

    https://www.cnblogs.com/madashu/p/12832766.html

    单个表的内数据如果随着时间的推移会变的越来越多。

    MYSQL单表：数据量300-500万。（而且需要做优化）。

    单个表的性能终究还是有限的。

    user user_info

    按业务逻辑拆分（时间、地区）

    mycat sharding-jdbc drds(阿里巴巴)

    总结：

    当前的设计（系统）架构能够保证高并发、高可用。（通过不断的对服务器进行扩容）

    随着业务和需求的不断变更和增加。（功能不断研发）同时 并发越来越高的情况下

    当前架构产生相应的问题。

    问题

    1、不断扩容服务器（服务器价钱昂贵、（忙闲不均））。服务器维护费用（场地+人工）大量运维工程师。

    2、可维护性差（代码、服务器维护部署）。

    3、可扩展性差（服务器、代码）

    4、协同开发不方便。（随着业务和需求的不断增加）【大家修改或操作的都是同一套业务代码】容易造成代码冲突）

    5、单体应用架构（所有的代码和业务需求都写在一个应用程序中）。这个架构代码会变的越来越臃肿。

    淘宝为例：（用户模块、支付模块、商品模块、购物车模块、订单模块、商户模块、图片资源管理、数据统计管理…..)。

    第二时期

    垂直应用架构

    当访问量逐渐增大，单一应用增加机器带来的加速度越来越小，提升效率的方法之一是将应用拆成互不相干的几个应用，以提升效率。此时，用于加速前端页面开发的Web框架(MVC)是关键。

    理解 ： 什么是垂直应用架构

    对架构进行拆分。

    将一个大的应用扩分成互不相干的多个小应用。

    水平拆分：

    按层拆分（MVC）

    问题1：现在网站分为前台访问（给用户该 问）和后台访问（管理员数据统计和数据维护用的）。

    垂直拆分(坚着拆）：

    按功能进行拆分

    将一个大的应用程序按功能拆分成多个互不相干的小应用程序。每个应用程序都是可独立部署（不同的服务器上）。

    解决：

    1、解决维护性（如果某个模块需求发生变更，只需要更改其模块就可）

    2、可扩展性（如果想新增加功能或需求，只需要新增新的模块即可）

    3、协同开发方便（不同的开发团队只需要负责相应自己功能模块即可)

    4、每个服务都是独立（并且再拆分）–》提高了服务器的运行效率、减轻了服务器的硬盘和内存的浪费占用等问题。

    阿里云、百度云、京东、新浪….

    问题：

    问题1：客户对页面的要求变的越来越高？（修改频繁）（需要重新启动服务器部署）？

    问题2：随着业务的不断增加（模块不断开发），服务之间的调用避不可少。如何解决服务间的调用问题？

    第三时期

    分布式服务架构

    当垂直应用越来越多，应用之间交互不可避免，将核心业务抽取出来，作为独立的服务，逐渐形成稳定的服务中心，使前端应用能更快速的响应多变的市场需求。此时，用于提高业务复用及整合的分布式服务框架(RPC)是关键。

    问题1：客户对页面的要求变的越来越高？（修改频繁）（需要重新启动服务器部署）？

    答：页面+业务代码（前后端分离部署）

    (使前端应用能更快速的响应多变的市场需求)

    问题2：随着业务的不断增加（模块不断开发），服务之间的应用之间交互不可避免。如何解决服务间的调用问题？

    总结：

    分析：

    将一个大的应用按功能拆分成互不相干多个小应用。（每个应用都是独立部署的）

    如果服务都在一个服务器内（可通过JAR依赖的方式即可完成相互调用)

    现在所有的服务都部署在不同的服务器（进程）。

    答：此时，用于提高业务复用及整合的分布式服务框架(RPC)是关键。

    解决方案：

    分布式服务框架（RPC)

    框架：

    进程和进程间完成服务调用技术：

    1、WebService –> Apche(CXF)–>Spring CXF整合。

    2、通过HTTP 方式完成访问

    3、RPC方式完成访问。 (RPC是远程过程调用（Remote Procedure Call）) Dubbo/apache thrift/grpc

    Dubbo是一个框架（RPC）框架。

    Dubbo是 一个分布式服务框架（解决服务间调用的问题）

    SpringCloud是一个分布式服务框架（HTTP)
